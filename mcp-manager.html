Staff Designer · Visor / Usercentrics · 2025–Present

MCP Manager

Role

Staff Designer (IC) · Sole designer

Problem

No tools for governing AI agent access at the org level

Approach

First-principles design, internal validation, alpha customer program, rapid iteration

Outcome

Acquired by Usercentrics (Jan 2026); in active use across a growing customer base

The situation

A new category of problem, no existing playbook, a market moving fast.

In early 2025, as MCP was emerging as an industry standard, Visor pivoted to build what organizations would actually need: governance infrastructure for AI agent access at scale. The question wasn't how to use MCP; it was how to manage it when dozens of teams need secure, auditable access to dozens of servers. I designed the product from scratch, leveraging the existing Visor design system as a foundation to move fast without starting over.

The challenge wasn't technical. It was trust. Who approved this connection? Which teams can reach which servers? What happens when an agent acts unexpectedly? I designed for two users who want opposite things: administrators who need full visibility and control, and developers who need MCP Manager to be invisible. That tension shaped every major decision.

I was the sole designer working without established conventions, in a fast-moving market where competitors were emerging in real time. No comparable tool to benchmark against, no mature research to draw from. Validation came from internal dogfooding, rapid iteration, and direct access to early customers through a formal alpha program.

The Work

Designing the mental model

Information architecture: making the invisible legible

The first design challenge was conceptual, not visual. MCP infrastructure is abstract: servers, gateways, agents, connections, policies. These relationships are opaque for most users. The product's first job was to make them visible and understandable before a user ever touches a configuration screen.

Welcome screen: concept, not yet shipped

The first screen a new user sees shouldn't be a settings page. It should be a map. The Welcome screen concept shows how the system works before asking anyone to configure it: AI apps on the left, gateways in the middle routing access, MCP servers on the right, users anchored below. Build the mental model first. Then everything else makes sense.

Welcome screen — entity relationship diagram (concept)

Navigation: four layers of concern

The sidebar IA reflects the two-user model across four layers: a Welcome onboarding area above the main nav; the action layer (Reporting, Alerts); the infrastructure layer (MCP Servers, Gateways, Apps & Agents); the governance layer (People, Policies, Logs); and the account layer. This keeps the surfaces developers care about at the top and the controls administrators depend on directly below, without requiring separate interfaces for each role.

Phase 1 nav — sidebar IA, four-layer structure

The gateway model: one prompt, many servers

The most important thing a gateway enables isn't architectural; it's experiential. A developer sends a single prompt, and the response can draw from multiple MCP servers simultaneously. Jira and Salesforce and Notion in a single answer, without the developer configuring each connection individually or knowing which servers were involved. From their perspective, it just works.

A gateway acts as the orchestration layer between AI applications and MCP servers. An AI app connects once, and through that single connection accesses every server assigned to it. The configuration burden that would otherwise fall on every individual developer is handled once, at the gateway level, by the administrator.

Local connections: one tunnel, one active server

Local MCP servers require tunnels to connect. To keep this manageable, a grouping construct ties a tunnel to a user's machine rather than to a specific server. At any given moment, one MCP server is active through it, reducing what would otherwise be an unmanageable number of connections to display and configure in the UI.

The System

Extending an existing design system to cover new territory.

MCP Manager was built on top of the existing Visor component library, a deliberate decision that accelerated delivery significantly. Rather than starting a design system from scratch, we extended the existing token and component architecture to cover MCP-specific UI patterns.

Brand direction: AI-assisted concepting

Visual brand exploration for MCP Manager was done using v0, my first foray into AI-assisted design. I used it to generate and pressure-test directions quickly, then brought the strongest concepts into Figma to finish the detail work. The resulting colors and typography are intentionally distinct from Visor and have been defined in the Tokens file. They haven't been applied to the product yet; velocity has taken priority, but the direction is set.

Design system — Components overview

Component evolution: annotation chips

Tool annotation chips (communicating properties like Read-only, Destructive, Idempotent, Open world) needed to represent three distinct epistemic states: known-true (solid filled chip), known-false (light background fill), and unknown (dashed outline + ? icon). This isn't a visual tweak; it's a meaningful semantic extension. Each state carries different implications for how an administrator interprets a tool's behavior and risk profile.

Annotation chips — three-state system (known-true, known-false, unknown)

Component evolution: breadcrumb navigation

As the product grew to support gateway-scoped views of MCP servers, a new global navigation pattern emerged. The breadcrumb + page header block became the standard wayfinding system for administrators navigating deep configuration hierarchies.

Evolution

From Phase 1 to Phase 3: how the product grew

  • Phase 1 (Sept 2025): Initial designs. Core IA, MCP Servers, Gateways, Apps & Agents, Users, Logs.
  • Phase 2 (Sept 2025): Sitemap × Roles. Visual design updates, Reporting and Alerts, permissions model, role-based access controls.
  • Phase 3 (Feb 2026): Managed. Added support for managed server type. Regex and Microsoft Presidio-based data rules replaced the earlier Policies model.
  • Phase 4 (Feb 2026, in progress): Workstations. New workstation server type. Design concepts for annotations-based tool provisioning.

The clearest example of evolution: condition building

The annotation chip system and the condition builder aren't two separate workstreams. They're the same work. The component evolution made the product capability possible, and the product capability is what gives the component its meaning.

Phase 1 condition building was attribute-based. The 'Add to list of conditions' popover offered three matching criteria: Tool name, Tool title, Tool description, functioning as a security gate. If an attribute didn't match a defined condition, the tool wasn't provisioned.

Phase 3 condition building is annotation-aware. The same popover now exposes the full annotation model: for each annotation type, an administrator can match any state, or target specifically true, false, or unknown. A rule like 'allow tools where Destructive is false' is categorically different from 'allow tools named X': it's semantic governance rather than label matching.

Outcome

Acquisition and what comes next

MCP Manager was acquired by Usercentrics in January 2026. The founding team integrated into a larger platform organization, and I continue as Staff Designer, now responsible for design operations and team integration in addition to the core product work.

The work is ongoing. The MCP ecosystem is still young, conventions are still forming, and the product will evolve as customers surface friction we haven't yet encountered.

Current focus: integrating MCP Manager operations with Usercentrics Legal and Sales teams, collaborating with PM on an MCPM pilot across Usercentrics, conducting competitive analysis, and designing an AI platform and MCP server registry for the next phase of the product.
