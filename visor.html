Visor: View Orientation

Role

Head of Product Design · Lead → Principal → Head

Problem

Users had no orientation to start with; a problem nobody had named yet

Approach

Introduced a status and config bar that evolved into a full orientation system and app architecture

Outcome

4× improvement on activation metrics

The situation

I noticed something no one had asked me to fix, and turned it into a product initiative.

Visor is a project and roadmap management platform built around bidirectional sync with tools like Jira and Asana. I joined in 2021 as Lead Designer, built the design system from scratch, and owned product design end-to-end for four years, progressing from Lead to Head of Product Design.

This case study is about one project within that tenure: the View Orientation Bar. I noticed a pattern in user research: people kept losing their place when switching between views. Visor lets users work across Table, Gantt, Board, Timeline, and Dashboard views, each with its own configuration state: active filters, sort order, visible fields. Set a filter, switch views, come back. Nothing tells you what you're looking at or why. The friction was real and consistent. No one had named it as a problem yet.

This wasn't a feature request and it wasn't on any roadmap. Advocating for something no one had asked for, in a fast-moving early-stage team, meant making the problem legible before I could ask for time to solve it. That required connecting observations across many research sessions into a coherent case.

The Work

View Orientation Bar → Visor Orientation Bonanza

A bar becomes an architecture

The first version was narrow: a persistent bar surfacing active filters, hierarchy depth, and record count. Simple, always visible, grounding. Early iterations tested how much information to surface and how to compress it, the key insight being that '3 Filters Applied' as a count badge was more scannable than a toggle or a column header icon.

As the bar proved its value, the question shifted from what it should show to what it should do. View settings that had lived in scattered menus consolidated into the bar. Then view creation moved in. Then the view name became a navigation entry point: a dropdown into a full Views & Folders panel showing every view across every collaborator, grouped by owner. The bar had absorbed what was previously scattered across the product into a single, consistent interaction layer.

Separately, as the interface grew more complex, Mike wanted individual elements to have enough presence to hold their own. The Crystalmorphic direction (his term, based on how light refracts through crystals) emerged as the answer, starting with the gradient canvas as the foundational visual element.

The shipped VOB: five views, one consistent pattern

The final VOB adapts its vocabulary to each view type while maintaining consistent placement and interaction. Table: Filters, Rows displayed, Groups. Gantt: Date fields, Filters, Bars plotted, Bar colors, Groups. Board: Columns, Lanes, Filters, Cards plotted, Card colors. Timeline: Date fields, Filters, Bars, plus edge case indicators like '19 Tasks Not Plotted'. Dashboard has no VOB, a deliberate IA decision reflecting its distinct role as a reporting canvas.

The concept that didn't ship, but then did

Late in the project, I explored a breadcrumb model for the top chrome: Workbook → Folder → View, all named and navigable, with the VOB stripped back to just Levels, Filter, and Records. The concept was deprioritized in Visor, not abandoned, but parted out. Elements fed into other areas of the product rather than shipping as a unified system.

The breadcrumb pattern shipped in MCP Manager, where the same hierarchy model became the primary wayfinding system for administrators navigating deep configuration trees. The Visor exploration didn't go to waste; it just shipped somewhere else.

The System

Atomic design, built from scratch.

The Visor design system was built from near-zero using atomic design principles: tokens up through components, patterns, and templates. The system grew with the product over four years and eventually expanded to cover MCP Manager as well. The same token and component architecture spans both products, with separate variable collections (Visor: 101 variables, MCP Manager: 91 variables) and distinct brand, color, and typography treatments for each.

Scale

  • 200+ components with 1,000+ variants and properties
  • 60+ patterns
  • 25 ideal resolution page templates + 25 minimum supported resolution page templates
  • 280+ medium and 280+ small icons
  • 40 external brands represented via icons and logos

The Crystalmorphic layer

The Crystalmorphic direction (Mike's term for how light refracts through crystals) is implemented through dedicated Effect styles ('Crystalmorphic' and 'Crystalmorphic Container' shadow tokens) combined with a gradient color system built on spectrum-spanning ramps. The gradient canvas is the most visible expression, but the depth and glass quality come from the shadow layer.

Five files

The system is split across five Figma files: Brands, Tokens, Icons, Components, and Templates, a necessary division at this scale. The files are linked and reference each other via shared token variables.

Outcome

4× improvement on the metrics that matter.

The VOB project's impact was measured through activation metrics, specifically the 'bought in' moment: registration → exploring features → inputting real data → refining a data story → inviting another user.

4×

Improvement in time to 'aha moment', faster activation from registration to real engagement

↑ Substantially

Weekly % of new users inviting another user, the retention signal that mattered most

The project also changed how the team evaluated subsequent feature work. A self-initiated problem definition that started with a bar displaying filter state ended with a new app architecture, a new visual language, and a measurable improvement in the metric that predicted long-term retention.
